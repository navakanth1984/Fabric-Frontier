# ACDLC Release Notes — v1.7

Transition from a **Distributed Agent Runtime (v1.6)** to a full **Agent Operating System Core**, introducing identity, trust, reputation, weighted routing, distributed worker pools, and execution tracing as first-class OS-level subsystems.

## New Subsystems

### Identity Service (`identity/`)

Introduces five modules forming the OS-level agent identity layer:

- **`registry.py`**: In-memory agent profile store keyed by `agent_id`. Publishes `AGENT_REGISTERED` / `AGENT_DEREGISTERED` events.
- **`profile_manager.py`**: Constructs and validates `AgentProfile` dicts matching `schemas/agent-profile.json` including `agent_id` pattern enforcement.
- **`trust_engine.py`**: Computes hard-clamped trust scores `[0.0, 1.0]` using `max(0.0, min(1.0, raw))` to prevent runaway values.
- **`reputation_engine.py`**: Tracks rolling success/failure/violation history per agent. Record violations automatically feed trust penalties into `TrustEngine`.
- **`quota_manager.py`**: Enforces per-agent resource quotas: `daily_token_budget`, `max_concurrent_tasks`, `max_retry_budget`.

### Weighted Routing Engine (`runtime/routing_engine.py`)

Replaces the v1.6 capability-exists check with a policy-driven multi-factor scoring formula:

```
score = (capability_weight × competency_score)
      + (trust_weight       × trust_score)
      + (load_weight        × (1.0 − current_load))
```

Weights are loaded from `policies/routing-weights.yaml` — not hardcoded. Default values:

```yaml
capability_weight: 0.5
trust_weight: 0.3
load_weight: 0.2
```

Saturation fallthrough: agents at or above `saturation_threshold: 0.85` are deprioritised and the engine falls through to the next-best candidate, implementing load-balancing override.

### Distributed Worker Pools (`workers/`)

Five role-isolated pool classes replacing the single `ExecutionWorker`:

| Pool | Priority | Capacity |
|---|---|---|
| `PlannerPool` | P2 | 4 |
| `BuilderPool` | P2 | 6 |
| `ReviewerPool` | **P1** | 3 |
| `OptimizerPool` | P2 | 3 |
| `RecoveryPool` | **P0** | 2 |

`RecoveryPool` is intentionally isolated on P0 so self-healing tasks never compete with normal execution workloads.

### Execution Trace Service (`analytics/traces/`)

Adds relationship tracking to the telemetry warehouse via `schemas/trace-event.json`:

```json
{
  "trace_id": "trace_abc123",
  "parent_event_id": "evt_001",
  "child_event_id": "evt_002",
  "agent_id": "builder_001",
  "session_id": "session_xyz",
  "routing_decision_id": "evt_route_001",
  "span_type": "child"
}
```

Enables: execution replay, root-cause analysis, latency tracing, distributed debugging, routing quality audit.

## Schema Upgrades

### `schemas/telemetry-event.json`
Added three optional v1.7 fields:
- `agent_id`: Identity of the executing agent
- `tenant_id`: Multi-tenant execution context (default: `"default"`)
- `trace_id`: Links event to parent execution trace

### `runtime/event_bus.py` (`Event` class)
Extended constructor to accept optional `agent_id`, `tenant_id`, `trace_id` parameters.

### `schemas/trace-event.json` (new)
Full parent-child trace correlation schema with `session_id` and `routing_decision_id` for forensic analysis.

---

## Policy Additions

### `policies/routing-weights.yaml` (new)
Externalises routing formula weights for policy-driven scheduling. Modifying this file changes routing behaviour without touching runtime code.

## Simulation Suite

### SIM-011: Weighted Routing Decision (new)
Validates the routing engine scoring formula:
- Two agents: `builder_a` (coding: expert, trust: high) vs `builder_b` (coding: advanced, trust: standard)
- Assert `builder_a` selected when loads are equal
- Assert `builder_b` selected when `builder_a` is saturated (load ≥ 0.85)
